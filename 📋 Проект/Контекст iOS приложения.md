# Контекст iOS приложения

Источник контекста: соседний проект `../MetrotimeIOS`, документация `raw/sources`.

## Назначение

МетроТайм - приложение для сотрудников Московского метро: машинистов, ТЧМ и других ролей.

Основные задачи:
- учет рабочих смен;
- расчет заработка, бонусов, надбавок и алиментов;
- статистика, месячная норма часов, годовые графики;
- синхронизация между устройствами.

## Что нужно перенести в web-версию

### Обязательная функциональность

- Авторизация пользователя.
- Список смен с группировкой по месяцам.
- Добавление, редактирование и удаление смен.
- Заметки к сменам.
- Экран текущей/ближайшей смены.
- Расчет часов: дневные, вечерние, ночные.
- Расчет заработка по тарифу и коэффициентам.
- Флаги смен: выходной, больничный, отпуск, резерв, наставник, ночная смена, ТЧМ, праздник, ИО СМ, старший машинист, перегон, профсоюз, алименты и другие.
- Экран нормы за месяц.
- Статистика за год.
- Настройки профиля, роли, тарифа, стажа, квалификации, наставничества, профсоюза и алиментов.

### Отличия web-версии

- Доступ через браузер, без App Store.
- Мобильный web-first интерфейс.
- PWA: можно добавить на главный экран.
- Премиум-подписка через web-платежи.
- Не завязана на WatchConnectivity, WidgetKit и App Group.

## Основная модель Shift

Ключевая сущность web-версии - смена.

Поля, которые нужно поддержать:
- `id`;
- `date`;
- `name`;
- `startTime`;
- `endTime`;
- `adjustedStartTime`;
- `adjustedEndTime`;
- `startPoint`;
- `endPoint`;
- `baseEarnings`;
- `mentorBonus`;
- `totalEarnings`;
- `breakBetweenShifts`;
- `nextShiftStartTime`;
- `notes`;
- набор boolean-флагов смены.

## Расчеты

Фактическая логика из `Shift.calculateHours()`:

| Тип часов | Диапазон | Коэффициент |
|---|---:|---:|
| Дневные | 06:00-16:00 | 1.0 |
| Вечерние | 16:00-22:00 | 1.2 для машиниста, 1.0 для ТЧМ |
| Ночные | 00:00-06:00 | 1.4 |

Важно: ранняя документация iOS местами указывала другой диапазон вечерних часов. Для web считать источником правды текущий код `Shift.swift` и `NormView.swift`.

## Риски, которые нельзя повторить бездумно

- В iOS Firestore хранит весь массив смен одной JSON-строкой. Для web лучше проектировать нормальную структуру таблиц/документов.
- В iOS роли хранятся строками. В web лучше сразу enum.
- В iOS заметки частично отделены от смен. В web лучше решить модель явно.
- Расчеты критичны, их нужно покрывать тестами с самого начала.
- Firebase/платежные ключи нельзя хранить на клиенте.
