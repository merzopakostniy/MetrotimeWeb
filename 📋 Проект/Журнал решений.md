# Журнал решений

Здесь фиксируем решения, чтобы не терять контекст.

## 2026-04-26

### Web-версия как отдельный продукт

Решение: MetrotimeWeb делается как мобильная web/PWA-версия iOS-приложения, чтобы доступ был у всех пользователей через браузер.

Причина: не ограничиваться iOS и App Store, а также получить возможность встроить web-подписки Premium.

### Obsidian как память проекта

Решение: весь контекст, требования, идеи, баги, архитектурные решения и платежные решения хранить в Obsidian vault `MetrotimeWeb`.

Правило: если появляется важная договоренность или изменение курса, сначала или сразу после работы фиксировать ее здесь.

### iOS-приложение как источник требований

Решение: ориентироваться на `../MetrotimeIOS` как на функциональный референс.

Важно: не копировать технические долги iOS бездумно. Для web сразу проектировать безопасное хранение, enum-роли, тестируемые расчеты и серверную обработку платежей.

### Источник правды для расчетов

Решение: если старая документация и текущий Swift-код расходятся, источником правды считать текущие исходники iOS.

Зафиксированное расхождение: диапазоны часов в реальном `Shift.calculateHours()`:
- день: 06:00-16:00;
- вечер: 16:00-22:00;
- ночь: 22:00-06:00.

### Одна смена на календарный день

Решение: зафиксировать iOS-поведение как текущее бизнес-правило: `upsertShift` удаляет все смены за тот же календарный день и добавляет новую.

Открытый вопрос: подтвердить, нужно ли сохранять это ограничение в web-версии или разрешить несколько смен в один день.

## 2026-04-26 (сессия 5)

### Фикс отображения профиля

Проблема: bottom sheet открывался но был пустым — контент не отображался.

Причина: `.ps-screen` использовал `position: absolute; inset: 0`, но родитель `.profile-sheet` не имел явной высоты — дочерние абсолютные элементы схлопывались.

Решение: задать `.profile-sheet { height: 92dvh }` и `.ps-screen { top: 24px }` (ниже drag-handle). SW → v13.

### Аватар-варианты для Сотрудника СБ

Решение: роль «Сотрудник СБ» переведена на `hasVariant: true` — добавлен picker м/ж как у машиниста и дежурного.

Файлы: `assets/avatars/avatar_sec_m.png` и `avatar_sec_f.png` (добавляются вручную из чата).

Старый файл `avatar_sec.png` более не используется — можно удалить.

SW → v14.

## 2026-04-26 (сессия 4)

### Выбор должности: двухэкранная навигация внутри профиля

Решение: «Должность» в настройках профиля — это nav-строка, по нажатию на которую внутри того же bottom sheet происходит слайд-переход на экран выбора роли. Никакого модала сверху, никаких выпадающих списков.

Технически: два `.ps-screen` внутри `.profile-sheet`. Основной экран (`#psMain`) сдвигается влево (`translateX(-30%)`) при переходе, экран выбора роли (`#psRoles`) выезжает справа (`translateX(0)`). CSS-transition 320ms cubic-bezier.

Экран выбора роли:
- Полноширинные карточки с PNG-аватаром и именем роли, как iOS `RoleWideCard`
- Для ролей с аватар-вариантами (`Машинист`, `Дежурный по станции`) снизу появляется picker с кружками «Мужской» / «Женский»
- Синяя кнопка «Сохранить» фиксирует выбор и возвращает на главный экран
- «Назад» (стрелка) возвращает без сохранения

Draft-паттерн: `roleDraft` хранит временный выбор в picker. При «Сохранить» применяется в `draft`. В профиль попадает только при финальном «Сохранить» на главном экране.

Аватары PNG из iOS-проекта скопированы в `assets/avatars/` (@2x из Xcode .imageset). При ошибке загрузки — emoji-заглушка.

Кнопка профиля в топбаре обновляется сразу после сохранения профиля (`updateTopbarAvatar()`).

SW-кэш: bumped до v12 при деплое этих изменений.

## 2026-04-26 (сессия 3)

### Экран профиля (bottom sheet)

Решение: профиль открывается как bottom sheet по нажатию на кнопку профиля в топбаре.

Структура данных профиля (localStorage, ключ `metrotime-profile-v1`):
```
role, avatarVariant (m/f), experienceDate, qualificationClass,
isSeniorDriver, isUnionMember, tariffRate, bonusPercentage,
alimonyChildren, tcmSalary, userId (генерируется автоматически)
```

Роли (из iOS): Машинист, Машинист-инструктор, Машинист ДДЭ, Служба безопасности, Дежурный по станции.

Аватары берутся из `assets/avatars/` — PNG-файлы из iOS проекта. До получения файлов — emoji-заглушки с цветом роли.

Логика стажа: 0-1 г. → 0%, 1-5 → 10%, 5-10 → 15%, 10+ → 20%+5% каждые 5 лет (макс 30%). Совпадает с iOS.

Кнопка профиля в топбаре отражает аватар текущей роли после сохранения.

Черновик (`draft`) не применяется до нажатия «Сохранить» — изменения не сохраняются при закрытии крестиком.

Ожидаем PNG-файлы: avatar_driver_m/f, avatar_tcm, avatar_dde, avatar_sec, avatar_station_m/f.

## 2026-04-26 (сессия 2)

### Полный перезапуск UI с нуля

Решение: старый прототип выброшен, UI переписан с чистого листа.

Причина: первая версия не подходила ни по структуре, ни по визуальному языку.

### Стартовый экран — Смены

Решение: при открытии приложения первым открывается экран **Смены**, а не Сегодня.

### Навигация: 6 вкладок с иконками и подписями

Итоговый состав нижнего таббара:
- **Смены** — иконка календаря
- **Сегодня** — иконка календаря с точкой (текущий день)
- **Норма** — иконка часов
- **Статистика** — иконка бар-чарта
- **13** — кастомная иконка: прямоугольник с цифрой внутри
- **Сообщения** — колокольчик (раздел пушей и уведомлений)

Решение: показывать иконку + подпись под ней (как iOS tabbar). Активная вкладка — голубой цвет `#3ecfff` + diffuse drop-shadow свечение вокруг иконки (через SVG `filter: drop-shadow`), без квадратных теней.

### Глобальный топбар

Решение: единый топбар на всех экранах вместо заголовка внутри каждого экрана.

Состав: кнопка профиля (слева) — название раздела по центру — кнопка + (справа). Заголовок меняется при переключении вкладок через JS.

### Деплой на GitHub Pages

Решение: деплой через GitHub Pages (ветка `main`, корень репо).

URL: https://merzopakostniy.github.io/MetrotimeWeb/

Важно: при каждом обновлении UI нужно менять версию кэша в `sw.js`, иначе сервис-воркер отдаёт старые файлы.

### Первый web-дизайн и бесплатный хостинг

Решение: стартовать с легкого статического mobile-first PWA-прототипа без зависимостей.

Причина: можно быстро собрать рабочий интерфейс, запустить локально и бесплатно разместить на Vercel, Netlify или GitHub Pages. Позже эту основу можно перенести в React/Next.js, сохранив структуру экранов и визуальный язык.

Первый экран должен быть не лендингом, а самим приложением: список/сводка смен, текущий месяц, быстрые действия, Premium-статус и нижняя навигация.
